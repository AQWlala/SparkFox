# RFC-002: 编排引擎协调机制

> **RFC 编号**: RFC-002
> **主题**: 蜂群 / DAG / Ralph / Tick 四种编排引擎的协调机制
> **状态**: 📝 草案（待评审）
> **创建日期**: 2026-07-18
> **评审专家**: 架构 / 性能 / 产品
> **影响 Phase**: Phase 1-3（必须在 Phase 0 决策）

---

## 一、背景与问题

### 1.1 原方案问题（架构专家评审）

原方案同时引入四种编排引擎，但未明确协调机制：

1. **谁是主编排？** 蜂群 / DAG / Ralph / Tick 谁是入口？
2. **冲突时谁让步？** 蜂群共识与 Ralph 永不放弃冲突时如何处理？
3. **Tick 心跳与 DAG 任务冲突？** Tick 在 DAG 执行中触发如何处理？
4. **性能问题**：性能专家警告 Tick 月成本 $22 + ACI 1.5s 同步阻塞 = UX 灾难

### 1.2 风险

- 四引擎打架 → 任务死锁或重复执行（架构专家 P0 风险）
- Tick 心跳成本失控（性能专家 P0 风险）
- ACI 1.5s 同步阻塞 → 用户感知卡顿（性能专家 P0 风险）

---

## 二、设计目标

1. **单一主编排**：明确一个主编排引擎，其他降级为策略插件
2. **冲突可解决**：定义冲突优先级与让步规则
3. **成本可控**：Tick 心跳可关闭，使用低成本模型
4. **ACI 异步化**：ACI 预判改为异步，不阻塞主循环

---

## 三、候选方案

### 方案 A: DAG 为主，其他为插件（推荐）

```
用户请求 → sparkfox-dag (主编排)
              ├─ 节点级 preflight
              ├─ 计划审批
              ├─ Worker 执行
              │   ├─ 调用 sparkfox-swarm（蜂群共识策略插件）
              │   ├─ 调用 sparkfox-ralph（永不放弃策略插件）
              │   └─ 调用 sparkfox-loop（反思策略插件）
              └─ 结果聚合

sparkfox-tick (独立后台)
  ├─ 空闲时触发（非任务执行中）
  ├─ 整理记忆 / 推进任务 / 检查提醒
  └─ 可由用户关闭

sparkfox-aci (异步预判)
  ├─ 消息到达时异步触发
  ├─ 1.5s 超时则放弃
  └─ 不阻塞主循环
```

**优点**：DAG 已有 react-flow 前端，可视化清晰；其他引擎降级为策略，无冲突
**缺点**：蜂群共识的"多 Advisor 投票"特性被弱化

### 方案 B: 蜂群为主

```
用户请求 → sparkfox-swarm (主编排)
  ├─ Advisor × 3 提议
  ├─ Orchestrator 协调
  ├─ Worker × N 执行（可调用 DAG 子图）
  └─ Verifier 验证
```

**优点**：蜂群共识是 Pangu 护城河
**缺点**：3 Advisor 投票成本高，简单任务过度编排

### 方案 C: 分场景路由

```
用户请求 → 路由器
  ├─ 简单任务 → 直接执行
  ├─ 复杂任务 → DAG 编排
  ├─ 创意任务 → 蜂群共识
  └─ 失败重试 → Ralph 接管
```

**优点**：每引擎发挥特长
**缺点**：路由器本身是新的编排引擎，复杂度反弹

---

## 四、推荐决策

**推荐方案 A**：DAG 为主，其他为策略插件

理由：
1. NomiFun DAG 已有 react-flow 前端，可视化成熟
2. 蜂群/Ralph/Loop 作为策略插件，可按需启用
3. 性能专家要求 Tick 异步化 + ACI 异步化，方案 A 天然支持

---

## 五、协调规则（待评审确认）

### 5.1 冲突优先级

| 场景 | 主引擎 | 策略插件 | 优先级 |
|------|--------|---------|:------:|
| 简单问答 | DAG 单节点 | 无 | DAG |
| 复杂任务分解 | DAG 多节点 | 蜂群（节点内共识） | DAG > 蜂群 |
| 节点执行失败 | DAG | Ralph 接管重试 | Ralph > DAG（失败时） |
| 反思整合 | DAG 完成后 | LoopEngine 触发 | DAG > Loop |
| 空闲整理 | Tick 后台 | 无 | Tick 独立 |
| 消息预判 | ACI 异步 | 无 | ACI 独立 |

### 5.2 Tick 心跳成本控制

- **默认关闭**：用户可选启用
- **低成本模型**：使用 GPT-4o-mini 或本地小模型，月成本 <$3
- **频率可调**：默认 30 分钟一次，用户可调
- **任务执行中不触发**：避免冲突

### 5.3 ACI 异步化

- **原方案**：1.5s 同步阻塞（性能专家 P0 风险）
- **优化后**：异步触发，1.5s 超时则放弃，不阻塞主循环
- **降级策略**：预判失败则按正常流程处理
