# RFC-003: 记忆体系单一真相源

> **RFC 编号**: RFC-003
> **主题**: Pangu 6 层 / OpenAkita 三层身份 / BaiLongma 线索焦点 三套记忆模型的单一真相源（SoT）设计
> **状态**: 📝 草案（待评审）
> **创建日期**: 2026-07-18
> **评审专家**: 记忆系统 / 架构 / 产品
> **影响 Phase**: Phase 1（必须前置决策）

---

## 一、背景与问题

### 1.1 原方案问题（记忆系统专家评审）

原方案引入三套记忆模型，但未明确单一真相源（SoT）：

1. **9 套机制重叠**：6 层 + 三层身份 + 线索/焦点 + 海绵/黑洞 + LoopEngine + 03:00 整合 + 04:00 自检 + MDRM + sqlite-vec，职责重叠严重
2. **身份层与 6 层关系不清**：OpenAkita SOUL.md/AGENT.md/USER.md/MEMORY.md 是独立存储还是 6 层的投影？
3. **L5 元认知定位问题**：L5 应为"横向平面"（跨层元数据）而非"第 6 层"（纵向叠加）
4. **遗忘机制完全缺失**（记忆系统专家 P0 级警告）

### 1.2 风险

- 三套模型冲突 → 记忆数据不一致（记忆系统专家 P0 风险）
- 9 套机制重叠 → 工程复杂度爆炸，开发周期失控
- 无遗忘机制 → 记忆膨胀 + 隐私合规失败（GDPR）

---

## 二、设计目标

1. **单一真相源**：明确一个 SoT，其他是投影或派生
2. **职责清晰**：6 层 = 认知深度，身份 = 价值观，线索 = 对话上下文
3. **L5 横向化**：L5 元认知改为跨层元数据平面
4. **遗忘机制**：定义遗忘策略（TTL / 容量上限 / 用户主动 / 合规擦除）

---

## 三、候选方案

### 方案 A: 6 层为 SoT，身份/线索为投影（推荐）

```
sparkfox-memory (SoT)
  ├─ L0 工作记忆（短期，会话级）
  ├─ L1 情景记忆（事件级）
  ├─ L2 叙事记忆（故事级）
  ├─ L3 语义记忆（事实级，向量检索）
  ├─ L4 程序记忆（技能级）
  └─ L5 元认知（横向平面，跨层元数据）

身份层文件（OpenAkita 投影）
  ├─ SOUL.md ← 从 L3 语义 + L5 元认知派生
  ├─ AGENT.md ← 从 L4 程序 + L5 元认知派生
  ├─ USER.md ← 从 L1 情景 + L3 语义派生
  └─ MEMORY.md ← 从 L0 工作记忆派生

线索/焦点（BaiLongma 投影）
  ├─ Threads ← 从 L0 + L1 派生（对话级线索）
  └─ FocusStack ← 从 L0 派生（当前焦点）

海绵/黑洞（写入路径）
  ├─ Sponge → 写入 L0/L1/L3
  └─ Blackhole → 压缩 L0/L1 → L2/L3

LoopEngine（反思路径）
  └─ 读 L0-L4 → 写 L5 元认知

03:00 整合 / 04:00 自检（维护任务）
  └─ 压缩 + 去重 + 修复 L0-L4
```

**优点**：6 层为 SoT，其他都是投影/派生，无冲突
**缺点**：身份层文件变为只读投影，用户编辑需同步回 6 层

### 方案 B: 身份层为 SoT，6 层为扩展

```
身份层文件（SoT）
  ├─ SOUL.md / AGENT.md / USER.md / MEMORY.md

6 层记忆（扩展存储）
  └─ 按身份层文件分类存储
```

**优点**：身份层透明可编辑
**缺点**：6 层的 L0-L5 纵向深度被弱化

### 方案 C: 双 SoT

```
身份层（价值观 SoT）
6 层（认知 SoT）
```

**优点**：各自独立
**缺点**：双 SoT 同步问题，记忆系统专家明确反对

---

## 四、推荐决策

**推荐方案 A**：6 层为 SoT，身份/线索为投影

理由：
1. 6 层是 Pangu 护城河，应为 SoT
2. 身份层作为投影，可读可编辑，编辑同步回 6 层
3. 记忆系统专家明确建议"身份层文件应为投影"

---

## 五、L5 元认知重新定位

### 5.1 原方案问题

原方案 L5 是"第 6 层"（纵向叠加），但记忆系统专家指出：

> L5 应为"横向平面"而非"第 6 层"。L5 是跨 L0-L4 的元数据（策略日志/错误模式/自评），不是与 L0-L4 并列的存储层。

### 5.2 优化后 L5 定位

```
L0-L4（纵向存储层）
  ↑
  │ 元数据
  │
L5（横向元认知平面）
  ├─ 策略日志：记录 L0-L4 的访问模式
  ├─ 错误模式：记录 L0-L4 的失败模式
  └─ 自评：对 L0-L4 质量的自评
```

L5 不存储原始记忆，只存储关于 L0-L4 的元数据。

## 六、遗忘机制设计（记忆系统专家 P0 要求）

### 6.1 遗忘策略

| 类型 | 策略 | 触发条件 |
|------|------|---------|
| **TTL 遗忘** | L0 工作记忆 24h 后自动归档 | 时间到 |
| **容量遗忘** | L3 语义记忆超过 10 万条时，低重要性自动压缩 | 容量上限 |
| **用户主动** | 用户可手动删除任意记忆 | 用户操作 |
| **合规擦除** | GDPR 被遗忘权请求 | 合规触发 |
| **黑洞压缩** | 低频访问记忆压缩为高维摘要 | 30 天未访问 |

### 6.2 遗忘审计

- 所有遗忘操作记录到 `memory_forget_log` 表
- 保留遗忘元数据（何时/为何/由谁触发）
- 支持 30 天内撤销（软删除）

## 七、9 套机制职责重划

| 机制 | 原职责 | 优化后职责 |
|------|--------|-----------|
| 6 层记忆 | 纵向存储 SoT | 纵向存储 SoT（不变） |
| 身份层文件 | 独立存储 | 6 层的投影（只读 + 编辑同步） |
| 线索/焦点 | 独立存储 | L0/L1 的投影（对话级） |
| 海绵引擎 | 吸收事实 | 写入路径（L0/L1/L3） |
| 黑洞引擎 | 压缩下层 | 维护路径（L0/L1 → L2/L3） |
| LoopEngine | 反思闭环 | 维护路径（读 L0-L4 → 写 L5） |
| 03:00 整合 | 日更整合 | 维护任务（去重 + 压缩） |
| 04:00 自检 | 日更自检 | 维护任务（错误修复） |
| MDRM 5 维度 | 关系图谱 | L3 语义记忆的关系索引 |
| sqlite-vec | 向量检索 | L3 语义记忆的向量索引 |

**职责分类**：
- 存储 SoT：6 层记忆
- 投影/派生：身份层 / 线索焦点 / MDRM
- 写入路径：海绵引擎
- 维护路径：黑洞 / LoopEngine / 03:00 / 04:00
- 检索索引：sqlite-vec / MDRM
